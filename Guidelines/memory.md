## HARD RULE (Non‑Negotiable) — Localize every new player-facing string

**Any time you add or change player-facing text, complete the full localization pipeline in the same change-set.**

1. **Author:** Use **`LOCTEXT` / `NSLOCTEXT`** or String Tables. Never `FText::FromString(TEXT("..."))` for shipping UI. No per-language `switch(CurrentLanguage)`.
2. **Gather:** Run GatherText so keys land in `Content/Localization/T66/`.
3. **Translate:** Populate all 22 cultures (e.g. `.archive` or `Scripts/AutoTranslateLocalizationArchives.py`).
4. **Compile:** Regenerate `Content/Localization/T66/<culture>/T66.locres`.
5. **Verify:** Confirm text changes when switching culture.

**Acknowledge in your next message:** e.g. “Acknowledged: new player-facing text will be localized (author → gather → translate → compile → verify).”

## Guardrails

- Atomic change-sets; ValidateFast after each.
- Small, descriptive commits.
- UI: **event-driven** only (no per-frame UI logic).
- **All UI text localized** via `UT66LocalizationSubsystem::GetText_*()` or NSLOCTEXT.
- **After C++ changes:** Close the editor, run the build command, then reopen. Do not use Live Coding / Hot Reload — the Config UObject `UT66RngTuningConfig` can trigger "Cannot replace existing object of a different class" (CDO replace crash) if the module is reloaded while the editor is running.
- **Sync load (main menu):** The only allowed sync load is in **UT66MainMenuScreen::BuildSlateUI()**: **UT66UITexturePoolSubsystem::EnsureTexturesLoadedSync()** is called for main menu + Party Picker textures (MMDark, MMLight, T_LB_Global, T_LB_Friends, T_LB_Streamers, SoloDark, SoloLight, CoopDark, CoopLight) so the background, leaderboard filter icons, and Party Picker cards appear immediately with no white flash. This is a safe "loading window" (frontend, not gameplay); no other sync texture loads in UI.
- **Level transitions:** When transitioning to GameplayLevel from the frontend (Enter Tribulation, Load Game), use **`UT66GameInstance::TransitionToGameplayLevel()`** instead of raw `UGameplayStatics::OpenLevel`. This shows a loading screen, pre-loads heavy assets, then opens the level. In-game stage reloads (StageGate, CowardiceGate, etc.) can still use raw `OpenLevel` since assets are already warm.

## Architecture (summary)

- **Core:** `T66GameInstance` (hero/companion/data tables), `T66AchievementsSubsystem` (profile, AC, save), `T66SkinSubsystem` (all skin API), `T66CharacterVisualSubsystem` (meshes/anim), `T66LocalizationSubsystem`, `T66RunStateSubsystem`, `T66SaveSubsystem`.

- **Flow:** Frontend (MainMenu → PartySizePicker → [Solo: HeroSelection | Duo/Trio New Game: Lobby → HeroSelection] → CompanionSelection → Enter) → GameplayLevel. Run state in RunStateSubsystem; death → RunSummary.

## Recent context (for continuity)

- **Casino replaces gambler/vendor NPCs:** Added `AT66CasinoInteractable` at the main-map center with a shared `UT66CasinoOverlayWidget` hosting 3 tabs: Gambling, Vendor, and Alchemy. Gambling/vendor behavior stays the same but both now feed one shared casino anger bar via `UT66RunStateSubsystem::AddCasinoAngerFromGold()` / `GetCasinoAnger01()`. Max anger now spawns `AT66GamblerBoss` only through `AT66PlayerController::TriggerCasinoBossIfAngry()`. Corner-house NPC spawn no longer includes gambler/vendor. Alchemy supports drag-and-drop target/sacrifice slots, upgrades the target item by one rarity tier, consumes the sacrifice item, and adds shared casino anger. New casino strings were gathered and `.locres` recompiled; non-English archives were backfilled with source-text fallback because `Scripts/AutoTranslateLocalizationArchives.py` is still missing.

- **Main menu button interior fill:** Main menu buttons now support a dedicated `ET66ButtonBackgroundVisual::MainMenuArcane` fill layered under the existing retro trim border and button body, without changing button size. `FT66Style::MakeButton()` now resolves optional state-driven fill brushes (Normal/Hovered/Pressed), and `UT66MainMenuScreen` opts into that fill for the large left-side menu buttons. Source generator: `Scripts/GenerateUIMainMenuArcaneFillTextures.py` → `SourceAssets/UI/MainMenuArcaneFill/T_UI_MainMenuArcaneFill_{N,H,P}.png`. Import script: `Scripts/ImportUIMainMenuArcaneFillTextures.py` → `/Game/UI/Assets/T_UI_MainMenuArcaneFill_{N,H,P}`. Runtime prefers imported `.uasset` textures, but falls back to file-backed Slate brushes from `SourceAssets/UI/MainMenuArcaneFill/` if the import has not been run yet. Clean editor restart still required before running `UnrealEditor-Cmd` import/build because the live editor session triggers the existing `T66UIManager` CDO/hot-reload crash.
- **Shared arcane UI defaults:** `FT66Style` now treats `ET66ButtonBorderVisual::Default` / `ET66ButtonBackgroundVisual::Default` as theme-driven defaults instead of “old dark chrome,” with explicit `None` opt-outs available. The shared defaults currently resolve to `RetroWood` + `MainMenuArcane` for buttons, and `MakePanel()` now applies the same arcane fill theme plus decorative trim for content panels (while `Bg` panels default to fill without the heavy trim). `UT66MainMenuScreen` no longer hardcodes those visuals; anything routed through `FT66Style::MakeButton()` / `FT66Style::MakePanel()` now inherits the central arcane theme automatically.
- **Arcane panel crash fix:** The first global panel-theme pass caused a Slate `EXCEPTION_ACCESS_VIOLATION` because `FT66Style::MakePanel()` handed `SBorder` / `SImage` raw `FSlateBrush*` pointers owned by local `TSharedPtr` brush sets. Fixed by switching the panel fill and decorative border brushes to attribute-backed lambdas that capture the shared brush sets, keeping them alive for the widget lifetime.
- **Default UI font reset to pixel font:** `FT66Style` default font index is back to `Alagard` (`GThemeFontIndex = 5`) instead of `Ransom`, so new UI rebuilds use the prior pixel-font look by default. `T66NextFont` still cycles through the full font list.

- **Import pipeline updated:** `Scripts/ImportStaticMeshes.py` now imports GLBs, flattens Interchange output, and immediately runs the GLB unlit conversion; `Scripts/ImportSkeletalMeshes.py` now imports FBXs and immediately runs the character unlit conversion. `Scripts/MakeGLBImportsUnlit.py` and `Scripts/MakeCharacterMaterialsUnlit.py` remain repair/re-run tools. `Scripts/SetupCharacterVisualsDataTable.py` reloads `DT_CharacterVisuals`; `Scripts/SetupItemsDataTable.py` reloads `DT_Items` after item CSV changes.
- **GLB atlas fidelity fix:** Some gameplay GLBs looked correct in Blender but muddy in Unreal. The durable fix lives in the GLB import/repair path: preserve source mesh data (`use_full_precision_u_vs = true`, `generate_lightmap_u_vs = false`, `recompute_normals = false`, `recompute_tangents = false`) and keep any atlas-specific texture overrides in the manifest. Important nuance: the mesh-build write requires `StaticMeshEditorSubsystem`, so run those repairs through **`UnrealEditor.exe -ExecutePythonScript=...`**, not `UnrealEditor-Cmd.exe`. Commandlets remain acceptable for material-only GLB repair, but not for build-setting mutation.
- **GLB stale-subasset root cause fixed:** clean raw imports of `Tree2.glb` and `Rock.glb` produced correct textures, but reimporting into the live gameplay folders had been leaving old nested `Texture2D` / `MaterialInstance` assets behind while only the flat mesh package refreshed. That was the real cause of the Blender-good / Unreal-wrong mismatch. `Scripts/ImportStaticMeshes.py` now performs pre-import cleanup per manifest entry. Default cleanup deletes the flat gameplay asset plus its same-named subtree; loot bags use `cleanup = {"mode": "dest_dir"}` because their folders are asset-specific.
- **GLB debug tools hardened:** `Scripts/DebugGLBRawImport.py` is the clean control test for a raw Interchange import, and `Scripts/ExportLightingIssueTextures.py` now exports the current mesh-bound `BaseColorTexture` instead of relying on hardcoded texture asset names that can go stale across reimports.
- **Prop grounding fix:** world props now follow the same grounding pattern already used by gates/interactables/NPC visuals: `FT66VisualUtil::GroundMeshToActorOrigin()` runs before `SnapToGround()` in `UT66PropSubsystem::SpawnPropsForStage()`. `Tree2` no longer needs a special `PlacementZOffset`; its `Props.csv` row is back to `0.0`.
- **Legacy slash disc removed from runtime dependency:** nothing active in C++ was spawning `AT66SlashEffect`, but its CDO still hard-loaded the deleted `/Game/VFX/M_SlashDisc`. The class now emits a short `NS_PixelParticle` ring instead, so startup/commandlet runs no longer depend on that removed material.
- **Fresh prop reimport utility:** `Scripts/RunImportPropsAndExit.py` now wraps `ImportStaticMeshes.py` and executes only the `Props/` manifest entries through `UnrealEditor.exe`, then quits. The current props batch now points at every GLB staged under `SourceAssets/Import/Props/` so the clean rerun path covers both same-name replacements (`Barn`, `Tree`, `Tree2`, `Bush`, `Boulder`) and the newly added prop set (`Branch`, `Fence*`, `Haybell`, `House`, `Log`, `Mud`, `Rocks`, `Scarecrow`, `Silo`, `Stump`, `Tractor`, `Tree3`, `Troth`, `Windmill`).
- **Import batch verification added:** `Scripts/VerifyImportBatch.py` validates the current prop / Hero 1 / companion batch and writes `Saved/Logs/VerifyImportBatch.json`. `Scripts/RepairStaticMeshImportBatch.py` re-runs the GLB repair pass without reimporting. - **Sprite size driven from C++:** All VFX use `NC->SetVariableVec2(FName("User.SpriteSize"), FVector2D(W, H))` to control pixel size.
