# T66 Master Guidelines

This file is the canonical project policy for engineering, content, and UI work in T66. When a rule here conflicts with an older note, this file wins.

## 1. Scope

- T66 is a UE5 C++ project.
- Runtime UI is built in Slate/C++; do not assume a UMG Designer workflow.
- Changes should target production-ready structures, not disposable prototypes.

## 2. Repository Structure Rules

- Keep the Unreal project root limited to required project roots, tooling, and top-level docs folders.
- Root-level documentation should live under `Docs/`, not beside the `.uproject`.
- Build/runtime dependencies such as `SourceAssets/`, `ThirdParty/`, `Scripts/`, `Source/`, `Content/`, `Config/`, and `Plugins/` stay at the project root unless intentionally refactored.
- Generated or scratch outputs should live under `Tools/`, `Saved/`, or a dedicated archive folder, never as random root clutter.

## 3. Unreal Root Rules

- Required stable root items:
  - `T66.uproject`
  - `.git/`, `.gitignore`, `.gitattributes`, `.vsconfig`
  - `Config/`, `Content/`, `Plugins/`, `Scripts/`, `Source/`, `SourceAssets/`, `ThirdParty/`
  - `Docs/`, `Tools/`
- `steam_appid.txt` remains at root while Steam local/dev flows depend on it.
- `Binaries/`, `Intermediate/`, `DerivedDataCache/`, and parts of `Saved/` are generated by Unreal and may reappear after builds or editor sessions.

## 4. Prime Engineering Rules

- Build the final structure first. Do not land throwaway systems that are expected to be replaced immediately.
- Prefer deterministic, schema-first changes.
- Keep low-end performance as a first-class requirement.
- Favor atomic change sets that can be validated clearly.
- Do not make broad destructive cleanup changes without first confirming the dependency surface.

## 5. Localization Rule

- Every new player-facing string must be localized at implementation time.
- Do not leave raw fallback English strings in shipping UI or gameplay flows except where the localization system already defines the fallback behavior.
- Treat localization as part of "done", not follow-up work.

## 6. Data / CSV / Content Rules

- Use project data under `Content/Data/` as the live source for CSV-backed systems.
- Keep schema changes explicit and compatible.
- Prefer source-of-truth data that can be regenerated or reimported cleanly.
- If a doc or scratch CSV exists outside `Content/Data/`, it is not live gameplay data unless explicitly documented as such.

## 7. Scripts, SourceAssets, and ThirdParty

- `Scripts/` is the home for editor automation, import scripts, and reproducible asset-generation workflows.
- `SourceAssets/` contains source art, fonts, generated source textures, and other runtime/editor fallback inputs used by code and scripts.
- `ThirdParty/` contains external runtime/build dependencies and is not a generic storage folder.
- When a new `.uasset` can be reproducibly generated, prefer a tracked script as the source of truth.

## 8. Slate and UI Framework Rules

- Prefer shared UI primitives and style helpers over ad hoc one-off widget composition.
- New button-like controls should use the shared button path, not bespoke `SButton` styling.
- New panel/surface shells should use the shared panel path, not raw `SBorder` stacks unless there is a strong reason.
- Do not introduce screen-specific scaling systems when a shared layout/scaling rule exists.
- New UI should be built against the current shared style/tokens layer in `FT66Style`.

## 9. UI Scaling and Resolution Policy

- Minimum safe layout baseline: `1280x720`.
- Primary handheld target: Steam Deck `1280x800`.
- Required validation targets:
  - `1280x720`
  - `1280x800`
  - `1366x768`
  - `1920x1080`
  - `2560x1440`
  - `3440x1440`
  - `3840x2160`
- DPI rule: `Shortest Side`.
- Critical UI should stay inside a centered safe frame; ultrawide side gutters are for decorative or secondary content, not required interaction.
- UI scale is the combination of project DPI scaling plus the player UI scale multiplier.
- Test UI in Standalone Game, not arbitrary dragged PIE viewport sizes.

## 10. Text Safety Rules

- No button label may clip.
- No panel body text may clip without either wrapping, scrolling, or deliberate truncation behavior.
- Buttons should grow to fit content before text is scaled down or clipped.
- Compact labels may ellipsize only when the UX explicitly calls for it.
- Text handling must be intentional:
  - wrap long body text
  - keep CTA text readable
  - avoid fixed shells that are narrower than localized text

## 11. Theme, Tokens, and Style Rules

- Layout-facing constants belong in shared tokens, not scattered raw literals.
- Theme/style code should define colors, brushes, fonts, icon sizes, padding, spacing, and safe shared geometry.
- Do not hardcode per-screen font sizes, padding, or slot sizes if the value is intended to be reused.
- Prefer named constants over repeated literal widths/heights.
- Optional texture/brush caches must keep their resources alive safely; do not store stale raw UObject pointers in static brush caches.

## 12. Layout Rules

- Critical screens and HUD clusters should be composed around the shared safe frame.
- Menus, modals, and commerce overlays should prefer content-driven or token-driven sizing over arbitrary fixed pixel shells.
- Use scrolling for overflow where the user must still access all content at the minimum safe layout.
- On large and ultrawide displays:
  - keep important interaction close to the core region
  - cap readable text widths
  - avoid pushing key actions to extreme edges

## 13. UI Anti-Patterns

- Do not tune UI by eyeballing arbitrary PIE viewport panel sizes.
- Do not add bespoke per-screen viewport scale math when shared DPI/safe-frame rules exist.
- Do not use rigid fixed-size buttons or panels that can clip localized or dynamic text.
- Do not introduce raw top-level docs and scratch files into the project root.
- Do not move `SourceAssets/` or `ThirdParty/` casually; both are part of active workflows.

## 14. Performance Rules

- Avoid widgets doing heavy work every frame.
- Avoid excessive widget churn from recreating large widget trees repeatedly.
- Keep UI effects, materials, and texture usage low-end friendly.
- Prefer event-driven UI updates over per-tick polling where possible.

## 15. Validation Rules

- For gameplay/UI code, validate with at least a compile/build gate before considering the change complete.
- For visual UI work, validate against the required resolution matrix in Standalone.
- When changing import pipelines or generated assets, verify the script path and the resulting runtime/editor behavior.

## 16. Documentation Rules

- System-level changes should update this file if they introduce or change project policy.
- Historical notes, migration context, and superseded decisions belong in `T66_DECISION_LOG.md`, not here.
- Detailed process docs should remain separate when they serve as references:
  - import pipeline
  - project catalogue
  - audits
  - implementation plans

## 17. Current UI Framework Decisions

- `ST66Button` and `ST66Panel` are the preferred shared Slate primitives for new work.
- `FT66Style` is the shared entry point for style/tokens/layout helpers.
- Shared scaling policy should be one system, not multiple per-screen heuristics.
- The current UI baseline is `1280x720` minimum-safe layout with UE DPI scaling and player UI scale layered on top.
