# T66 Codebase Diagnostic Report â€” March 31, 2026

Comprehensive audit of 236 `.cpp` and 275 `.h` files across `Source\T66\`.

---

## CRITICAL â€” Crash Risks & Bugs

### 1. Unchecked `GetGameInstance()` null dereference (~25 call sites)

Many files use this pattern that will crash if `GetGameInstance()` returns null during level transitions or teardown:

```cpp
// DANGEROUS: guards GetWorld() but not GetGameInstance()
UT66RunStateSubsystem* RS = GetWorld() ? GetWorld()->GetGameInstance()->GetSubsystem<UT66RunStateSubsystem>() : nullptr;
```

**Affected files:**
- `T66PowerUpScreen.cpp`
- `T66RunSummaryScreen.cpp`
- `T66WheelOverlayWidget.cpp`
- `T66CollectorOverlayWidget.cpp`
- `T66PlayerController_Overlays.cpp`
- `T66PlayerController.cpp`
- `T66PlayerController_Input.cpp`

**Fix:** Guard both calls: `GetWorld() && GetWorld()->GetGameInstance() ? ... : nullptr;`

---

### 2. `IsSecondaryPercent` logic divergence â€” display bug

In `T66StatsPanelSlate.cpp`, the standalone `IsSecondaryPercent()` function (line 96) correctly covers 9 stat types including `ReflectDamage` and `CounterAttack`. But `MakeEssentialStatsPanelFromSnapshot` (line 598) has a hardcoded inline copy that is **missing both**. ReflectDamage and CounterAttack show as raw floats instead of percentages on the leaderboard run summary screen.

---

### 3. Hard-coded `Hero_1` override bypasses DataTable

In `T66GameInstance.cpp` (lines 372â€“376), `GetHeroData` has a special case:

```cpp
if (HeroID == FName(TEXT("Hero_1")))
{
    OutHeroData.UltimateType = ET66UltimateType::ScopedSniper;
}
```

This silently overrides DataTable data and will break if someone edits the DataTable thinking it's authoritative.

## HIGH â€” Performance Issues

### 4. `SaveSubsystem::LoadOrCreateIndex` does synchronous disk I/O on every query

`FindFirstEmptySlot`, `FindOldestOccupiedSlot`, `GetSlotMeta`, and `UpdateIndexOnSave` each call `LoadOrCreateIndex()`, which does a synchronous `UGameplayStatics::LoadGameFromSlot` every time. The index should be cached in memory after the first load.

---

### 5. `PowerUpSubsystem::Save` uses synchronous save

Uses `UGameplayStatics::SaveGameToSlot` (synchronous) while `SaveSubsystem` correctly uses `AsyncSaveGameToSlot`. Every unlock or crystal spend blocks the game thread on disk I/O.

---

### 6. `RunStateSubsystem` is a 4,200+ line God Object

This single subsystem manages HP, gold, debt, inventory, vendor shop, buyback, idol delegation, hero stats (primary + secondary + category-specific), all combat multipliers, DOT tracking, 15+ passives, survival mechanics, difficulty scaling, stage timers, boss state, event logging, luck rating, and more. It should be split into at least 3â€“4 focused subsystems.

- **28 dynamic multicast delegates** at the top, some firing 17 simultaneously during `ResetForNewRun`
- **~150 lines of pure pass-through** to `IdolManagerSubsystem` that should be removed
- **Two ~35-case switch statements** (`GetSecondaryStatValue` / `GetSecondaryStatBaselineValue`) that are nearly identical â€” should be data-driven

---

### 7. `GameplayHUDWidget` is a 5,300-line monolith

Handles health, gold, boss bar, cooldowns, inventory strip, idol slots, pickup cards, tutorial hints, crosshair, scoped sniper, quick revive, status effects, curse overlay, full map, minimap, FPS counter, NPC dialogue, TikTok placeholder, wheel spin, crate open, achievement notifications, and debug buttons â€” all in one class with **70+ cached `TSharedPtr<>` members**.

`NativeTick` calls `RefreshCooldownBar()` and `RefreshSpeedRunTimers()` **every single frame** regardless of whether values changed.

---

### 8. Boss `Tick` calls `GetPlayerPawn` every frame (uncached)

`AT66BossBase::Tick` calls `UGameplayStatics::GetPlayerPawn(this, 0)` every frame, while `AT66EnemyBase` correctly caches this in a `TWeakObjectPtr<APawn>`.

---

### 9. Per-enemy tick overhead: 5 manual countdown timers

Every alive enemy decrements `ArmorDebuffSecondsRemaining`, `MoveSlowSecondsRemaining`, `ForcedRunAwaySecondsRemaining`, `ConfusionSecondsRemaining`, and `AutoAttackKnockbackSecondsRemaining` every frame. With hundreds of enemies, this is `5 * N` float ops per frame. Should use `FTimerHandle` callbacks.

---

### 10. `GetIdolColor` â€” 39 sequential FName comparisons

15 legacy alias remappings + 24 color lookups, all via sequential `if (IdolID == FName(...))` creating temporary FName objects on every call. Replace with a static `TMap<FName, FLinearColor>`.

---

### 11. `VendorStockItemIDs.Contains()` in nested loop

During vendor stock generation, `TArray::Contains()` (O(n)) is called in a nested loop for every template Ă— 3 stock slots. A `TSet` would be O(1).

---

### 12. `HandleInteractPressed` â€” 18-way Cast chain on every overlap result

Performs `OverlapMultiByChannel` with a 300-unit sphere, then iterates all results with 18 separate `Cast<>` calls in an if-else chain. Should use an `IT66Interactable` interface with virtual `GetInteractionPriority()` and `Interact()`.

## HIGH â€” Massive Code Duplication

### 19. `CircusOverlayWidget` and `CasinoOverlayWidget` are near-identical clones

1,033 vs 1,031 lines for .cpp, 107 vs 108 for .h. The only differences are class name prefixes. This is **~2,100 lines of duplicated code**.

---

### 20. Inventory strip duplicated across 4 overlay widgets

`VendorOverlayWidget`, `GamblerOverlayWidget`, `CircusOverlayWidget`, and `CasinoOverlayWidget` all declare identical inventory slot arrays, sell panels, buyback arrays, and refresh functions. A shared `ST66NPCInventoryStrip` would eliminate this.

---

### 21. StatsPanelSlate: 3 near-identical builder functions

`MakeEssentialStatsPanel`, `MakeLiveEssentialStatsPanel`, and `MakeEssentialStatsPanelFromSnapshot` repeat the same 8 primary stat lines and secondary category iteration. Only the data source differs. A single parameterized builder would cut ~300 lines.

---

### 22. Hero/Companion Selection structural duplication

Both implement identical 5-slot carousels, skin panels, lore toggles, language handlers, grid launchers, and preview stage integration. A shared base class could extract ~600 lines.

---

### 23. Armor debuff system copy-pasted between `EnemyBase` and `BossBase`

Identical `ApplyArmorDebuff()` and `GetEffectiveArmor()` implementations with identical member variables. Should be extracted to a shared component.

---

### 24. 9 identical `Resolve*Class()` functions

All follow the exact same pattern: check `IsDotaTheme()`, return Dota override or `StaticClass()`. A single template helper would replace all 9:

```cpp
template<typename T>
TSubclassOf<T> ResolveThemeClass(TSubclassOf<T> DotaOverride) const
{
    if (FT66Style::IsDotaTheme() && DotaOverride)
        return DotaOverride;
    return T::StaticClass();
}
```

### 25. 101-header include block copied across 7 PlayerController partials

Each `T66PlayerController_*.cpp` starts with the same ~101 includes. A shared header or PCH would reduce compile time and maintenance.

## MEDIUM â€” Memory & Safety

### 34. `WebImageCache::Waiters` has no lifetime tracking

Callers register `TFunction<void(UTexture2D*)>` callbacks that capture `this` pointers, but there's no mechanism to remove stale callbacks if the requesting object is destroyed before download completes.

---

### 35. `ActiveDOTs` uses `TWeakObjectPtr` keys in `TMap`

UE docs warn against weak pointers as map keys due to hash instability.

---

### 36. Global static textures never cleaned up

`GMinimapIconAtlas` in `T66GameplayHUDWidget.cpp` calls `AddToRoot()` with no corresponding `RemoveFromRoot()`. `GTopBarFileTextureCache` in `T66FrontendTopBarWidget.cpp` accumulates entries and is never cleared outside of `WITH_EDITOR`.

---

### 37. Screen cache never prunes

Once created, screens stay in `ScreenCache` forever. For 20+ screen types, this keeps all widget trees alive in memory.

---

### 38. Missing object pooling

Boss projectiles, ground AOE effects, Hero_1 attack VFX actors, and loot bag pickups are all spawned/destroyed frequently but not pooled (while enemies and floating combat text correctly are).
